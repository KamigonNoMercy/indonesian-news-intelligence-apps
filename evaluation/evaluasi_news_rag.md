# News RAG System Evaluation

This document summarizes how the evaluation was carried out and its results,
measured per category, per sentiment, and overall. The evaluation was run on 100
question-answer interactions (generated by `batch_run.py`, logged to
`rag_logbook.jsonl`, then scored by `evaluate_logbook.py`).

Evaluation date: 14 June 2026.

---

## Evaluation Method

The evaluation is **automatic** and **reference-free** (no answer key / gold set).
The idea: another LLM acts as a "judge" (LLM-as-a-judge) that scores the system's
answer based on the question and the context used.

This approach is **similar to RAGAS** - we use the same metrics (faithfulness and
answer relevancy) with the same LLM-as-a-judge principle. The difference is that
full RAGAS requires many API calls (and a separate embedding step), which is token
-expensive. Due to API token limits, we run the evaluation using **Groq API only**
as the judge - the main API we use in this project - with the model
`llama-3.1-8b-instant`, which has a more generous daily budget. (The RAGAS path is
still available in the code via the `--ragas` option, but it is not used as the
primary evaluation for quota reasons.)

The flow:

1. **Generate** - 100 questions are run through the RAG pipeline. The questions are
   *grounded* to the corpus content (balanced across categories and sentiments),
   plus 12 *out-of-corpus* questions (outside the data coverage) to test whether
   the system honestly says "not found".
2. **Log** - each interaction (question, answer, context, category, sentiment) is
   recorded one line per interaction to `rag_logbook.jsonl`.
3. **Judge** - each interaction is scored by the judge model (`llama-3.1-8b-instant`
   via Groq). Faithfulness and relevancy are asked in a single call to save tokens.
   Results are saved to `logbook_eval.csv` (per-interaction detail) and
   `evaluation_summary.csv` (aggregate).
4. **Incremental** - already-scored interactions are automatically skipped, so the
   evaluation can be re-run without spending tokens on the same items.

## Metrics Used

| Metric | What it answers | Range | Measured on |
|--------|-----------------|-------|-------------|
| **Faithfulness** | Are all claims in the answer actually supported by the context? (1 = grounded, 0 = fabricated) | 0 - 1 | Answered interactions |
| **Answer Relevancy** | Does the answer actually address the question? (1 = highly relevant, 0 = off-topic) | 0 - 1 | Answered interactions |
| **Anti-hallucination** | For "not found" answers, was the refusal correct (context truly irrelevant)? | 0 - 1 | Refused interactions |

Note: faithfulness and relevancy are only computed on **answered** interactions;
anti-hallucination is only computed on **refused** interactions ("not found"). A
`-` in the table means there is no sample for that metric in that group (e.g. no
refusals, so anti-hallucination is undefined).

---

## Overall Results

| Scope | n | Answered | Refused | Faithfulness | Relevancy | Anti-hallucination |
|-------|---|----------|---------|--------------|-----------|--------------------|
| Baseline (no filter) | 67 | 60 | 7 | 0.677 | 0.703 | 1.000 |
| All (+ filtered) | 100 | 91 | 9 | 0.630 | 0.652 | 1.000 |

**Anti-hallucination 1.000** in every scenario: whenever the system answered "not
found", the refusal was indeed correct (context not relevant). This means the
system did not fabricate answers for out-of-coverage questions - its strongest
point.

|----------|---|----------|---------|--------------|-----------|--------------------|
| finance | 22 | 22 | 0 | 0.595 | 0.641 | - |

| news | 17 | 16 | 1 | 0.588 | 0.594 | 1.000 |
| sport | 21 | 18 | 3 | 0.667 | 0.722 | 1.000 |

**Quick read:**

- **health** is the strongest (faithfulness 0.758, relevancy 0.753) - the most
  grounded and most relevant answers.
- **inet** is the weakest (0.525 / 0.525) and also the most refused (4 out of 20).
  Likely a thinner corpus for this category or more specific questions.
- **finance** answered every question (0 refusals) but scored mid-range, so there
  is no anti-hallucination value for this group.
- **sport** has high relevancy (0.722) despite mid-range faithfulness.

## Results by Sentiment

This sentiment grouping only covers questions that were given a **sentiment filter**
(11 questions per sentiment). The `no-filter` row is for questions without a filter,
identical to the overall baseline, shown as a comparison.

| Sentiment | n | Answered | Refused | Faithfulness | Relevancy | Anti-hallucination |
|-----------|---|----------|---------|--------------|-----------|--------------------|
| negative | 11 | 11 | 0 | **0.682** | **0.736** | - |
| neutral | 11 | 10 | 1 | 0.460 | 0.490 | 1.000 |
| positive | 11 | 10 | 1 | 0.460 | 0.410 | 1.000 |
| no-filter (comparison) | 67 | 60 | 7 | 0.677 | 0.703 | 1.000 |

**Quick read:**

- **negative** is the strongest (faithfulness 0.682, relevancy 0.736), and it even
  answered every question. This makes sense: negatively-toned news (outbreaks,
  conflicts, accidents) tends to be fact-dense and explicit, so it is easier to
  ground.
- **positive** and **neutral** are clearly weaker (faithfulness 0.460).
  Positive/neutral-toned questions tend to be more general/abstract, so the answer
  is harder to tie directly to a source sentence.
- When questions are **constrained by a sentiment filter**, average scores drop
  versus the baseline - consistent with the overall faithfulness dropping from
  0.677 (baseline) to 0.630 (all + filtered), because the filter narrows the pool
  of candidate context that can be retrieved.

---

## Summary

- **Perfect anti-hallucination (1.000)** across all groups that had refusals: the
  system does not fabricate for out-of-data questions. This is the main target of a
  *grounded* RAG design, and it is achieved.
- **Mid-range faithfulness & relevancy (~0.63 overall)**: answers are generally
  grounded and relevant, with the most room to improve in the **inet** category and
  **positive/neutral** sentiment.
- **Filters narrow the context**: enabling category/sentiment filters slightly
  lowers faithfulness/relevancy because fewer context candidates are available - a
  reasonable trade-off between filter precision and context completeness.
